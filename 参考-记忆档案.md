## 🏗️ 决策优先级（层越低越优先）

```
L0: 用户说的 > L1: 安全底线 > L2: 工程纪律 > L3: 工作流程 > L4: 代码哲学 > L5: 质量门禁
```

哪层能解决问题就停在哪层，不往下走。

### L0 — 用户优先
用户直接说的需求，永远第一优先级。技能规则不能高于用户意图。

### L1 — 安全底线（碰都不要碰）
- 凭证不写代码/Git，统一 `os.environ.get()`
- 涉及钱/手机验证/封闭平台/他人实名 — 不碰
- 服务器：不改药店页面、不改防火墙规则、不做不可逆操作
- 工具设计用 🟢自动/🟡人工判断/🔴禁止 三级模型

### L2 — 工程纪律（动手前默背）
1. 改前先读 — 服务器代码改前先 `cat` 全文
2. 改后验语法 — `python -m py_compile`
3. 部署验存活 — `curl` 确认 HTTP 200
4. 一次一处 — 改一处验一处，不一口气改三处
5. 每次留记录 — 改了什么、结果写进 events/

### L3 — 工作流程（大功能走完整，小改动走轻量）

**大功能/新项目**：
```
先搜GitHub → 写Spec → 确认 → 实现 → 测试 → 部署 → 记录
```

**小改动/修bug**：
```
确认问题 → 直接修 → 验证 → 记录
```

### L4 — 代码哲学（Ponytail 七级阶梯）
每写一行代码前问自己，停在第一个能站稳的台阶：
```
1. 这东西真的需要写吗？        (YAGNI)
2. 代码库里已有现成的？        (复用)
3. 标准库自带？                (标准库)
4. 平台原生支持？（html标签等） (原生)
5. 已装依赖能解决？            (已有依赖)
6. 能不能一行搞定？            (一行)
7. 写最小可工作版本            (最小实现)
```

### L5 — 质量门禁
- **neat-freak 轻量收尾**：每次会话结束检查 代码/文档/记忆 是否一致，小项目不做
- **触发词覆盖**：功能/命令覆盖中英文+口语多种说法
